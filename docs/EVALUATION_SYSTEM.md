# 增量开发测评体系 (Evaluation System)

在对 Agent 架构进行深度重构（如拆分子图、增加断点、引入自进化）时，必须保证现有的核心功能和基线成功率不发生退化。

为此，我们建立了一个**三级递进式测评体系 (3-Tier Evaluation Strategy)**，并将其固化在项目根目录的 `Makefile` 中。每改动一块内容，遵循以下测试流：

## 测评层级定义

### L1: 单元与逻辑边界测试 (Unit & Boundary Tests)
- **目标**：验证架构底层逻辑（FSM 流转规则、Hook 拦截、工具权限控制、上下文合并语义）的正确性。
- **⚠️ 维护提醒**：由于当前处于架构大重构阶段，**L1 测试本身也必须随着架构演进（如引入 Reducer、修改 LoopState 结构）同步更新**。如果只改架构不改测试，会导致大量 False Alarms（误报错误）。
- **触发时机**：每次修改 `core/*.py` 或 `hooks/*.py` 代码后，保存时立即执行。
- **耗时**：数秒。
- **执行命令**：`make test`

### L2: 迷你基准测评 (Mini Benchmark)
- **目标**：验证完整 Agent 工作流（Prompt 组装 -> LLM 推理 -> 沙箱工具执行 -> 解析回退）未被破坏。
- **机制**：从 VulRepair 靶场中抽取 1-2 个运行最稳定且成功率高的案例（如 `case_02_CWE_125`），在一个隔离的工作区下进行端到端运行。
- **触发时机**：完成一个逻辑块的修改后（如改完了一层 Context 截断），提交代码前执行。
- **耗时**：2-3 分钟（一杯咖啡的时间）。
- **执行命令**：`make eval-mini` 或 `make check`（组合执行 L1+L2）

### L3: 真实环境与全量防退化测评 (Full Regression & SEC-bench)
- **目标**：验证系统级指标未发生整体性退化，并挑战真实世界的高复杂度漏洞环境。
- **机制**：
  - **全量 VulRepair**：并发执行完整跑测（10个 Cases），并生成与历史对比的 `vulrepair_benchmark_report.json` 报告。
  - **真实环境 SEC-bench**：利用 `scripts/run_secbench_single_test.py` 拉取真实漏洞容器（如 `mruby.cve-2022-0240`），验证 Agent 在面对复杂项目构建（`make`, `gcc`）和真实 CWE 时的端到端能力。
- **触发时机**：完成一个完整 Phase 的架构升级（如完成了整个自进化体系）时，大版本演进验证。
- **耗时**：数十分钟至 1-2 小时。
- **执行命令**：
  - VulRepair 全量：`make eval-vulrepair-full`
  - SEC-bench 抽样：`make eval-secbench`

---

## 增量开发标准工作流 (SOP)

1. **开发与修改**：修改某一块内容（例如重构了 `transition_handler` 逻辑）。
2. **同步更新测试用例**：如果修改了核心数据结构，务必同步更新 `tests/` 下对应的单元测试。
3. **快速验证 (L1)**：终端运行 `make test`。如果失败，立即修复；如果成功，进入下一步。
4. **端到端验证 (L2)**：运行 `make check`。系统会自动先跑 `test`，接着隔离跑测一个具体的漏洞修复用例。
5. **深入挑战 (L3, 可选)**：如果是发版前，运行 `make eval-secbench`，确认项目能在 SEC-bench 真实容器中执行构建与漏洞验证。
6. **确认并迭代**：当测试全部通过时，说明这块改动是安全的。
