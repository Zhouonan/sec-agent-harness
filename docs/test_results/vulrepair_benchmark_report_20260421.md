# Sec-Agent-Harness: VulRepair 漏洞修复能力评测报告 (2026.04.21)

## 1. 评测概览
- **评测目标**：评估 Sec-Agent-Harness 在处理真实 C 语言漏洞（CWE 类型）时的分析、验证及修复能力。
- **评测数据集**：`VulRepair` 校准集（10 个典型案例，涵盖 Top-10 CWE 类型）。
- **运行环境**：FSM 驱动架构，`gcc:11` 沙箱，LLM 引擎（kimi-k2.5）。

## 2. 评测结果摘要
| 指标 | 结果 |
| :--- | :--- |
| **总用例数** | 10 |
| **成功完成 (DONE)** | 6 (60%) |
| **执行异常 (ERROR)** | 4 (40%) |
| **平均耗时** | 约 530 秒 / 用例 |
| **逻辑修复准确度** | 100% (在成功完成的用例中) |
| **完全字符匹配 (EM)** | 0% (预期的，受数据集格式影响) |

## 3. 详细指标分析

### 3.1 成功案例分析 (Success Cases)
在 **60%** 的成功用例中，Agent 展现了卓越的安全推理能力：
- **深度语义理解**：Agent 准确识别了 `vp8_cal_sad` (CWE-119) 中由于 `== 0` 导致的边界逃逸，并自主修改为更安全的 `<= 0`。
- **多步验证 (PoC 驱动)**：在 Case 10 (CVE-2017-9993) 中，Agent 成功编写了 C 语言 PoC，验证了 HTTP Cookie 在共享字典中的信息泄露路径 (CWE-200)。
- **安全补丁标准性**：Agent 采用的 `LONG_MAX / count` (CWE-190) 溢出检查方案符合工业级 C 语言编码规范 (MISRA-C)。

### 3.2 失败模式分析 (Failure Modes)
**40%** 的失败主要集中在以下两个维度：
1.  **步数溢出 (Max Turn Exhaustion)**：
    - Case 03 (IOAPIC 模拟器) 和 Case 08 (TIFF 库) 达到了 40 回合上限。
    - **原因**：Agent 陷入了复杂的沙箱编译环境配置循环，或者在 Review 阶段无法构建出能 100% 覆盖补丁逻辑的测试用例。
2.  **环境依赖缺失**：
    - 部分 C 库（如 `libavcodec`, `libtiff`）在默认 `gcc:11` 镜像中缺少头文件。Agent 花费了大量步数尝试 `apt-get install`，但由于没有 sudo 或网络限制而受阻。

## 4. 技术洞察
- **FSM 韧性**：`REVIEWER` 状态起到了核心质量把关作用。多次实验显示，如果 Patch 无法通过自建测试，Agent 会主动回退至 `FIXER` 重新修复，有效防止了低质量补丁上线。
- **隔离感评估**：在修改评测脚本隐藏绝对路径后，Agent 不再尝试回溯宿主机的 `/challenge` 目录，表现出良好的环境专注度。

## 5. 优化建议
1.  **资源限制调整**：对于 C 语言任务，建议将 `total_max_turns` 上调至 **60 轮**。
2.  **沙箱镜像增强**：构建一个 `sec-agent-base:v1` 镜像，预装常用的安全开发库（如 `zlib-dev`, `libssl-dev`, `libtiff-dev`）。
3.  **Prompt 纠偏**：在 `VALIDATOR` 阶段，增加对“最小化包装 main 函数”的指令引导，减少 Agent 调试 C 代码链接报错的时间。
