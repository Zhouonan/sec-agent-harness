# 架构博弈：状态枚举 (FSM) vs. 多智能体协作 (Multi-Agent)

在设计 `Sec-Agent-Harness` 时，我们面临一个核心选择：是让一个 Agent 通过 **状态枚举 (State Enum)** 切换身份，还是雇佣一群 **多智能体 (Multi-Agent)** 互相协作？

---

## 1. 方案定义

### A. 状态枚举 (FSM / 单智能体多态)
- **定义**：同一个 LLM 实例，在不同的任务阶段加载不同的 `System Prompt` 和工具集。
- **类比**：一个“全能特工”，根据任务需要，随时换上“医生”、“黑客”或“狙击手”的装备。

### B. 多智能体协作 (Multi-Agent / 异构团队)
- **定义**：存在多个独立的 LLM 实例（Agent），每个 Agent 只有单一职责，通过消息进行通信。
- **类比**：一个“特种部队”，分析师发现漏洞，写报告交给验证员，验证员跑完 PoC 再交给修复员。

---

## 2. 核心优劣对比

| 维度 | 状态枚举 (FSM) | 多智能体协作 (Multi-Agent) |
| :--- | :--- | :--- |
| **确定性 (Control)** | **极高**。流程由状态机硬编码，绝不会出现“分析还没做完就去修复”的情况。 | **较低**。Agent 间通信是自然语言，容易出现沟通误解、任务推诿或死循环。 |
| **上下文效率 (Context)** | **极佳**。通过“黑板机制”只传递核心结论。跳转状态时，旧的无效对话史可以直接丢弃。 | **一般**。跨智能体通信往往需要重复背景信息，容易导致上下文冗余。 |
| **实现难度** | **简单**。只需维护一个状态变量和一套跳转规则。 | **复杂**。需要设计复杂的通信协议（如 SOP）和协调者（Orchestrator）。 |
| **灵活性 (Scalability)** | **一般**。状态路径是预设的，难以应对突发的复杂支线任务。 | **极高**。可以动态增加审计员、法务等角色，适合处理非线性的超大规模任务。 |
| **Token 消耗** | **低**。精准控制 Prompt。 | **高**。多轮跨 Agent 交互非常烧 Token。 |

---

## 3. 为什么 Sec-Agent-Harness 选择 FSM？

在**安全审计与自动化修复**场景下，我们选择了 FSM 骨架，原因有三：

1. **防止“工具中毒” (Tool Hallucination)**：
   - 漏洞验证工具（如沙箱执行）非常危险且复杂。如果全部丢给一个 Multi-Agent 环境，分析阶段的 Agent 可能会误用验证工具，导致系统误报或资源浪费。FSM 保证了在 `INITIAL_ANALYSIS` 状态下，Agent **根本看不见**验证工具。

2. **状态流转的原子性**：
   - 漏洞发现 -> 验证 -> 修复 -> 审计，这是一个极其标准、线性的逻辑链。用状态机强制流转（SOP 数字化），比靠 Agent 互相聊天来推进流程要稳健得多。

3. **上下文压实 (Context Compaction)**：
   - 安全扫描的原始日志非常庞大。FSM 跳转时的“清空消息+黑板存档”逻辑（s06），是目前长周期 Agent 保持“智力不退化”的最有效手段。

---

## 4. 总结：给学习者的复习重点

- **FSM** 适合 **“深度、线性、高风险”** 的任务（如：修复一个特定的漏洞）。
- **Multi-Agent** 适合 **“广度、非线性、跨领域”** 的任务（如：为一个完整项目制定安全标准）。

**在你的简历项目 `AgentHarness` 中，强调 FSM 就是在强调你对“工程确定性”和“Token 优化”的极致追求。**
