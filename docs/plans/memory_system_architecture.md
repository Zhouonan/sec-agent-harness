# Agent 记忆架构演进与集成规划 (Memory Architecture Plan)

## 1. 行业前沿调研：三大主流 Memory 架构对比

随着 Agent 从“单次对话机器人”向“持久运行的协同工作者”演进，如何突破 LLM 的 Context Window（上下文窗口）瓶颈，成为了核心工程挑战。目前业界主要有三大流派：

### 1.1 MemGPT (操作系统分页流派)
* **核心理念**：将大模型的上下文当成电脑的“内存 (RAM)”，将向量数据库当成“硬盘 (Disk)”。
* **工作机制**：Agent 内部拥有操作系统的“分页（Paging）”权限。当上下文满了，Agent 会**自主调用** `archive_context` 或 `swap_out` 工具，将不重要的信息写回硬盘；当需要时，调用 `core_memory_append` 将硬盘里的信息调回主显存。
* **适用场景**：需要全自动、长时效（跑几个月）运行的自主探索型 Agent。

### 1.2 Mem0 (混合结构化流派)
* **核心理念**：放弃把原始对话扔进数据库，只提取“事实 (Facts)”。
* **工作机制**：当一段对话结束，Mem0 会通过大模型抽取这段对话里的关键实体和状态（例如：“用户在 /src 目录下发现了一个 XSS 漏洞”），然后支持对这些事实进行 `ADD`, `UPDATE`, `DELETE`。底层采用 Vector（查语义） + Key-Value（查确切属性） + Graph（查关系）的混合存储。
* **适用场景**：生产级、企业级的高效查表型 Agent。

### 1.3 Zep (时态知识图谱流派)
* **核心理念**：信息是有保质期的（Temporal Knowledge Graph）。
* **工作机制**：在存储信息的同时，加上时间戳和“有效期”。比如昨天代码里有个报错，今天修好了，图谱中“报错”这一节点就变成了“过去时”。当 Agent 提问时，不仅能查到事实，还能查到事实的演变轨迹。
* **适用场景**：复杂的长周期业务，如代码重构、跨越多周的渗透测试。

## 2. 结合 Sec-Agent-Harness 的现状分析

在目前的 `core/loop.py` 中，我们主要采用的是**“断舍离 + 黑板摘要”**的硬截断策略。
* **优势**：极度节省 Token，防止 Agent 在跨阶段时（如 ANALYSIS 到 FIXER）的注意力污染。
* **劣势**：**存在不可逆的失忆**。大模型一旦进入 FIXER 阶段，如果发现之前分析的漏洞路径有误想要回溯细节，它现在只能干瞪眼，因为原始 `messages` 已经被我们无情清空了，黑板上的 `summary` 又太简略。

## 3. 下一代 Memory 演进方案：混合型长短时记忆体系

为了弥补这一短板，我们计划在不破坏现有 FSM 高效隔离的前提下，引入以下混合记忆架构（借鉴 MemGPT 与 Mem0 的思想）：

### Phase 1: 外部记忆检索工具 (被动召回)

**目标**：打破 Agent 失去上下文后的失忆困境，赋予其类似 MemGPT 的分页检索能力。
**实现路径 (存储与检索)**：
1. **存储层设计 (Storage)**：
   * 拦截大模型每轮的原始通信：在 `core/loop.py` 每次完成 API 调用后，提取 `User Request`, `Tool Action`, `Tool Output`。
   * **结构化落盘**：将这些数据加上元数据（`turn_id`, `FSM_state`, `timestamp`）后，存入本地 **SQLite 数据库**（并开启 FTS5 扩展以支持高性能分词全文检索）。
   * **语义层 (可选)**：如果需要更模糊的匹配，可异步调用本地 embedding 模型，将记录存入轻量级的 ChromaDB。
2. **检索工具开发 (Retrieval)**：
   * 在系统 Tool Set 中新增一个工具 `search_episodic_memory(query: str, state_filter: str = None)`。
   * **触发与查询词构造 (Trigger & Query Generation)**：您可能会问，Agent 既然“失忆”了，怎么知道要搜什么？这就是 **黑板（Blackboard）** 的秒用。在 `FIXER` 阶段，Agent 的上下文里留有 `plan`（例如：“Step 3: Fix SQLi in db.py”）和 `ANALYSIS_summary`（例如：“在 db.query 中发现拼接注入”）。这些就是**记忆索骥的“线索”**。当它发现 summary 里只有结论而没有报错代码的细节时，它会自动提取这些线索词（构造 `query="db.query SQL injection"`）作为关键字调用工具。
   * **底层 Match 匹配逻辑 (How MATCH works)**：
     - **FTS5 虚拟表索引 (什么是虚拟表与分词？)**：
       * **痛点**：如果我们只是把日志存进普通的数据库表，当 Agent 想搜 `db.query` 时，数据库只能用 `LIKE '%db.query%'` 笨拙地从头到尾扫一遍几万字的文本，极其缓慢且无法做相关度排序。
       * **分词与倒排索引**：FTS5 是 SQLite 里的一个小巧的“搜索引擎”。在你存入报错日志的那一瞬间，它就会把长篇大论的文本**切成一个个独立的词汇**（这叫分词）。然后它建了一本“字典”（倒排索引）：记录着 `db` 这个词出现在第 3 条日志，`query` 出现在第 3、5 条日志。
       * **为什么叫“虚拟表”**：因为它在代码里看起来像一张普通的表（你可以对它写 `SELECT` 语句），但它底层并不是用来按行存数据的，而是专门为了存那本“分词字典”而设计的。当你搜索时，它是直接查字典秒出结果，而不是扫全表。
     - **BM25 相关度打分**：当大模型传入 `query` 时，底层的 `MATCH` 操作符使用的是经典的 **BM25 检索算法**。它不仅判断词在不在，还会计算**词频 (TF)** 和**逆文档频率 (IDF)**。比如 "db.query" 这个词在历史中只出现过一次，那么含有它的那条记录相关度得分就会极高。
     - **组合查询提纯**：SQL 层面会执行类似 `SELECT * FROM memory WHERE state = 'ANALYSIS' AND content MATCH 'db.query' ORDER BY rank LIMIT 3`。这保证了大模型用极短的 `query` 也能像用 Google 搜索一样，精准捞出潜藏在历史堆栈里的那段具体源码。
3. **业务效果**：当 Agent 在 `FIXER` 阶段忘记漏洞触发路径时，它可以调用该工具查询：“搜索 ANALYSIS 阶段扫描漏洞的详细输出”，将需要的“几页内存”调回主显存。

### Phase 2: 结构化事实图谱 (主动固化)

1. **记忆结晶机制 (独立的后台提取器)**：
   * **当前问题**：目前系统完全依赖主 Agent 在调用 `transition_state` 时自己费心去写 summary。这加重了主 Agent 的“认知负担”，且主 Agent 很容易因为格式拼错导致工具调用失败。
   * **进阶方案 (Independent Summarization Call)**：剥离总结职能！主 Agent 只需发出一个简单的转移指令。后端系统在拦截到转移指令后，**单独发起一次新的大模型 API 调用**（甚至可以专门配置一个更便宜、速度更快的小模型）。这个独立调用的 System Prompt 专门被设定为“你是一个无情的 JSON 抽取器”，它读取刚结束的冗长对话，严格输出 **JSON 格式的事实 (Fact)** 并挂载到黑板上。
2. **格式约定**：
   ```json
   {
     "vulnerabilities": [{"id": "v1", "type": "SQLi", "file": "db.py", "status": "identified"}],
     "dead_ends": ["尝试绕过 waf 失败，因为使用了 strict 模式"]
   }
   ```
3. **对比自然语言总结的绝对优势 (Why JSON over Prose?)**：
   * **防信息磨损与幻觉**：自然语言总结在经历多次状态流转的“总结再总结”后，细节极其容易丢失或变异。而 JSON 结构允许大模型像操作数据库一样**精确更新局部字段**（例如只把 `v1` 的 `status` 从 `identified` 改为 `patched`），绝不影响其他内容。
   * **Token 消耗严格受控**：人类的自然语言总结很容易越写越长，甚至包含大量口语化废话。结构化的 Schema 强制约束了表达形式，保证黑板区的 Token 占用始终保持在低水位。
   * **工程流转 (Programmability)**：解析出来的 JSON 列表，可以直接作为硬参数喂给其他的防御工具（比如拿去自动生成 WAF 规则，或自动给这些文件加权限锁），而一段自然语言文本是无法直接用于机器自动化处理的。

### 4. 实施优先级
* **High**: 修复目前代码中漏掉的 `messages` 截断逻辑（已在本次会话中完成！保障了现有的设计基线）。
* **Medium**: 引入 `search_episodic_memory` 工具，让历史日志不再只是给人看的，也能让大模型自己检索。
* **Low**: 全面引入 Graph 或 Vector DB，重构当前的 Blackboard。
